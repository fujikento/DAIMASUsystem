# 03 映像生成パイプライン設計書

## 概要

イマーシブダイニング・プロジェクションシステムの映像生成パイプライン設計。
物理テーブル（8120mm x 600mm）に3台プロジェクターで投影する5520x1200px（4.6:1）の超ワイド映像を、
AI画像/動画生成APIを組み合わせて効率的に制作する。

---

## A. 画像生成パイプライン

### 現状分析

現在の画像生成フローは以下の通り:

```
[プロンプト] → Imagen 4 Fast (16:9, 1408x768) → crop-to-fill → 5520x1200
                                                    ↓
                                              61%の情報ロス
```

- Imagen APIは最大16:9（1408x768）を出力
- crop-to-fill で 5520x1200 (4.6:1) にフィットすると、上下約61%の画素が切り捨てられる
- 中央の水平帯のみが最終出力に残り、構図の自由度が大幅に制限される

Geminiテンプレートガイド方式（現在の推奨）:

```
[テンプレート画像 1456x316] + [プロンプト]
    → Gemini 2.5 Flash Image (マルチモーダル)
    → crop-to-fill → 5520x1200
```

- テンプレート画像がレイアウトを誘導するため、4.6:1に近い構図が生成される
- crop時の情報ロスは30-40%に低減（Imagen比で大幅改善）
- 品質は現時点で最も高い

### 改善案1: 2-pass分割生成 + 合成

```
[プロンプト(左半分)] → Gemini/Imagen → 左パネル (2760x1200)
[プロンプト(右半分)] → Gemini/Imagen → 右パネル (2760x1200)
                                            ↓
                                    ブレンド合成 → 5520x1200
```

**手順:**
1. 基本プロンプトに「左半分 / 右半分」の位置指示を追加
2. 左右それぞれを16:9で生成（各1408x768）
3. crop-to-fill で各2760x1200にリサイズ
4. 中央200pxのオーバーラップ領域をクロスフェードブレンド
5. 最終的に5520x1200を合成

**トレードオフ:**
| 項目 | 評価 |
|------|------|
| 品質 | 中 - 左右の一貫性確保が課題。色味・スタイルの不一致リスク |
| コスト | 2倍 - API呼び出しが2回 |
| 速度 | 並列実行で単体とほぼ同等（5-8秒） |
| 情報ロス | 約40% - 各パネルのcrop量は低減されるが完全には解消しない |

**備考:** Geminiのシード値制御ができないため、左右の一貫性は確率的。
複数回生成して最も整合性の高いペアを選択する戦略が必要。

### 改善案2: Geminiテンプレートガイド最適化（現行ベスト）

```
[最適化テンプレート 1456x316] + [最適化プロンプト]
    → Gemini 2.5 Flash Image / Gemini 2.5 Flash Pro Image
    → crop-to-fill → 5520x1200
```

**最適化ポイント:**
1. テンプレート画像の改善
   - ゾーン分割線をより繊細に（生成画像への影響を最小化）
   - コンテンツエリアの枠線色をテーマカラーに合わせる
   - プレート位置マーカーのサイズ調整（現在やや大きい）
2. プロンプトの最適化
   - テーマ別に専用プロンプトテンプレートを用意（`prompt_templates.py`）
   - 「中央水平帯にコンテンツを集中」指示をより具体的に
   - カラーパレット指定による色味の安定化
3. モデル選択
   - `gemini-2.0-flash-exp-image-generation`: 高速・低コスト（プレビュー向き）
   - `gemini-2.5-flash-image`: 高品質（本番向き）

**トレードオフ:**
| 項目 | 評価 |
|------|------|
| 品質 | 高 - テンプレートがレイアウトを強く制御 |
| コスト | 1倍 - 単一API呼び出し |
| 速度 | 5-10秒 |
| 情報ロス | 30-40% - 4.6:1に近い構図が誘導されるがcropは必要 |

### 改善案3: Inpainting/Outpaintingによる拡張

```
[プロンプト] → Imagen 16:9 (1408x768) → 基本画像
    → 左方向outpaint (+700px) → 拡張画像
    → 右方向outpaint (+700px) → 拡張画像
    → crop-to-fill → 5520x1200
```

**手順:**
1. 中央部分を通常通り生成（16:9）
2. Imagen EditのOutpainting機能で左右を段階的に拡張
3. 最終的に4.6:1以上のキャンバスを構築
4. 5520x1200に最終フィット

**トレードオフ:**
| 項目 | 評価 |
|------|------|
| 品質 | 中-高 - outpaint境界でのアーティファクトリスク |
| コスト | 3倍以上 - 生成1回 + outpaint2回以上 |
| 速度 | 15-25秒（逐次処理のため） |
| 情報ロス | 10%以下 - ほぼ全域が有効コンテンツ |

**備考:** Imagen 4のEdit APIでoutpaintがサポートされているが、
繰り返しoutpaintでの品質劣化を検証する必要がある。

### 推奨戦略

**プレビュー（ストーリーボード）:** 改善案2（Geminiテンプレートガイド最適化）
- 速度重視。`prompt_templates.py` の最適化プロンプトで品質を底上げ

**本番映像のベース画像:** 改善案2 + 改善案3のハイブリッド
- Geminiテンプレートで高品質な基本画像を生成
- 左右のエッジが切れている場合のみoutpaintで補完

## B. 動画生成パイプライン

### Runway Gen-4.5 Turbo API

**API仕様（Image-to-Video モード）:**

```
POST https://api.dev.runwayml.com/v1/image_to_video
Authorization: Bearer {RUNWAY_API_KEY}

{
    "model": "gen4_turbo",
    "promptImage": "<base64 or URL>",
    "promptText": "motion description...",
    "duration": 10,
    "ratio": "16:9",
    "seed": 42
}
```

**レスポンスフロー:**
1. POST → タスクID返却（即座）
2. GET /v1/tasks/{id} でポーリング（10-60秒）
3. status: "SUCCEEDED" → output[0] に動画URL
4. URLからMP4をダウンロード

**制約:**
- 最大解像度: 1280x768 (16:9) / 768x1280 (9:16) / 768x768 (1:1)
- duration: 5秒 or 10秒
- 同時実行: 5タスクまで（Turboプラン）
- アスペクト比: 16:9, 9:16, 1:1 のみ（21:9は非対応）

### Image-to-Video ワークフロー

```
[生成済み画像 5520x1200]
    → 16:9にcrop (中央部 2133x1200 or リサイズ)
    → Runway i2v (10秒, 16:9)
    → 生成動画 1280x768
    → upscale + crop-to-fill → 5520x1200
    → touchdesigner/content/themes/{theme}/unified/
```

**プロンプト最適化:**
- Image-to-Videoでは `promptText` はモーション指示に特化
- シーン説明は画像が担うため、動きだけを記述
- 例: `"gentle wave-like motion flowing left to right, particles drifting, slow ambient movement"`

### 5コース x 各10秒 バッチ生成戦略

```
welcome  ─┐
appetizer ─┤
soup      ─┼→ 最大5並列 → Runway同時実行上限5に収まる
main      ─┤
dessert   ─┘
           ↓ 全完了後
     合成・後処理パイプライン
```

**バッチ実行フロー:**
1. 5コース分の画像を事前生成（Gemini/Imagen、10-15秒）
2. 5画像を同時にRunway i2vへ投入（並列、10秒duration指定）
3. 全タスクをポーリング監視（タイムアウト120秒）
4. 完了した動画を順次ダウンロード
5. 各動画をcrop-to-fill で 5520x1200 に後処理
6. `touchdesigner/content/themes/{theme}/unified/` に保存

**合計所要時間（推定）:**
- 画像生成: 15秒（並列）
- 動画生成: 60-90秒（Runway処理時間）
- 後処理: 10秒
- **合計: 約90-120秒で5コース分の動画が完成**

### ループ可能な動画生成

動画をシームレスにループさせるテクニック:

1. **プロンプトでのループ指示:**
   ```
   "seamless loop, the end frame matches the beginning,
    continuous cyclical motion, no abrupt transitions"
   ```

2. **後処理でのクロスフェード:**
   - 最後の1秒と最初の1秒をffmpegでクロスフェード
   ```
   ffmpeg -i input.mp4 -filter_complex
     "[0:v]split[body][pre];
      [pre]trim=start=0:end=1,setpts=PTS-STARTPTS[head];
      [body]trim=start=0:end=9,setpts=PTS-STARTPTS[main];
      [main][head]xfade=transition=fade:duration=1:offset=8"
     output_loop.mp4
   ```

3. **TouchDesigner側でのループ制御:**
   - Movie File In TOPの Loop パラメータを活用
   - 複数動画の切り替え時にフェードトランジションを適用

## C. コンテンツ合成パイプライン

### 3プロジェクター分割とエッジブレンド

**物理レイアウト:**
```
|←──── PJ1 (1920px) ────→|
                    |←120px→|
                    |←──── PJ2 (1920px) ────→|
                                        |←120px→|
                                        |←──── PJ3 (1920px) ────→|

|←─────────────── 5520px (有効投影幅) ──────────────────→|
```

**分割座標:**
| プロジェクター | X開始 | 幅 | X終了 |
|--------------|-------|------|-------|
| PJ1 | 0 | 1920 | 1920 |
| PJ2 | 1800 | 1920 | 3720 |
| PJ3 | 3600 | 1920 | 5520 |

**エッジブレンド考慮事項:**
- 120pxのオーバーラップ領域でグラデーションブレンドが必要
- ソフトウェアブレンド（ffmpegでアルファグラデーション適用）は `content_compositor.py` の `split_for_projectors()` が担当
- ハードウェアブレンド（TouchDesigner / プロジェクター内蔵）はリアルタイム合成時に適用
- ブレンドカーブ: ガンマ2.2補正済みリニアフェード推奨

### ゾーン間トランジションのプリレンダリング

コース切り替え時のトランジション映像を事前生成:

```
[コースA最終フレーム] → トランジション (3秒) → [コースB初期フレーム]
```

**トランジションパターン:**

| パターン | 内容 | 推奨テーマ |
|---------|------|----------|
| fade_black | 黒へフェードアウト→フェードイン | 全テーマ共通 |
| wave_wipe | 波形ワイプ（左→右） | ocean |
| flame_dissolve | 炎のディゾルブ | fire |
| petal_scatter | 花びら散り→集まり | zen, forest |
| particle_burst | パーティクル爆発 | space, gold |
| page_turn | ページめくり | fairytale |

**プリレンダリング戦略:**
1. 各コース間のトランジション（4パターン x 7テーマ = 28映像）
2. 解像度: 5520x1200、3秒、30fps
3. ffmpegで生成可能なものはffmpegで、複雑なものはRunway i2vで生成
4. `touchdesigner/content/transitions/{theme}/` に保存

### リアルタイムレイヤー合成

TouchDesigner上での合成レイヤー構造:

```
Layer 4: テキストオーバーレイ（料理名、メッセージ）
Layer 3: パーティクルエフェクト（リアルタイム生成）
Layer 2: トランジション映像（プリレンダリング済み）
Layer 1: ベース映像（AI生成動画）
Layer 0: 黒背景（フォールバック）
```

**合成ルール:**
- Layer 1 (ベース映像) は常時再生
- Layer 2 (トランジション) はコース切り替え時のみ3秒間アクティブ
- Layer 3 (パーティクル) はプロジェクションモードに応じて適用
- Layer 4 (テキスト) はOSC/WebSocketコマンドでトリガー

**パフォーマンス目標:**
- 合成レイテンシ: 16ms以下（60fps維持）
- GPU VRAM使用量: 2GB以下（5520x1200 x 4レイヤー）
- ディスクI/O: SSD前提、プリフェッチで動画切り替えを最適化

## パイプライン全体フロー

```
┌─────────────────────────────────────────────────────────┐
│  1. コンテンツ企画                                         │
│     テーマ選択 → コース設定 → ストーリーボード作成             │
└───────────────────────────┬─────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│  2. 画像生成 (プレビュー)                                   │
│     prompt_templates.py → Gemini/Imagen → 5520x1200 画像   │
│     所要時間: 5-10秒/枚                                     │
└───────────────────────────┬─────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│  3. 動画生成                                               │
│     生成画像 → Runway i2v (10秒) → crop → 5520x1200 動画    │
│     所要時間: 60-90秒/本 (5本並列で合計90-120秒)              │
└───────────────────────────┬─────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│  4. コンテンツ合成                                          │
│     動画 + トランジション + オーバーレイ → 最終映像             │
│     3PJ分割 → touchdesigner/content/ に配置                 │
└───────────────────────────┬─────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│  5. リアルタイム投影                                        │
│     TouchDesigner → OSC制御 → 3プロジェクター出力             │
└─────────────────────────────────────────────────────────┘
```
